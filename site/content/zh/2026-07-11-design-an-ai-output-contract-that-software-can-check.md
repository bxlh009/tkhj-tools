将大型语言模型（LLMs）集成到生产级软件中时，最常见的故障点并非模型的智能程度，而是其不可预测性。LLM 可能上一秒生成有效的 JSON 对象，下一秒却生成一个 Markdown 列表，或者遗漏必填字段。为了构建可靠的系统，开发人员必须将 AI 输出视为必须遵守严格契约的数据，而非自由形式的文本。

本文概述了如何设计“AI 输出契约”——即在生成之前定义的一组规则——使下游软件能够自动验证、拒绝或纠正 AI 响应。这种方法将可靠性的负担从事后解析转移到事前规范。

## 核心概念：预先规范

AI 输出契约本质上是模式（schema）与行为指令的结合。它定义了：
1.  **结构：** 包含哪些字段？（例如，JSON 键、XML 标签）。
2.  **类型：** 数据类型是什么？（例如，字符串、整数、布尔值）。
3.  **约束：** 允许哪些值？（例如，枚举值、正则表达式模式、范围）。
4.  **缺失数据行为：** 模型应如何处理不确定性？（例如，返回 `null`、返回 `"unknown"` 或拒绝回答）。
5.  **拒绝条件：** 何时应完全丢弃输出？（例如，如果触发安全过滤器，或置信度分数较低）。

通过在提示词中指定这些元素，您可以让软件以编程方式检查输出是否符合这些规则。如果输出违反契约，软件可以立即捕获错误，而不是在数据库插入或 API 传输期间稍后崩溃。

## 步骤 1：定义模式

任何契约的基础都是一个严格的模式。虽然许多模型支持原生 JSON 模式，但仅依赖“JSON 模式”对于复杂的业务逻辑是不够的。您必须在结构内定义*内容*约束。

例如，如果您正在从支持工单中提取客户意图，基本模式可能如下所示：

```json
{
  "type": "object",
  "properties": {
    "intent": {
      "type": "string",
      "enum": ["refund", "technical_support", "billing_inquiry", "other"]
    },
    "urgency": {
      "type": "integer",
      "minimum": 1,
      "maximum": 5
    },
    "summary": {
      "type": "string",
      "maxLength": 200
    }
  },
  "required": ["intent", "urgency", "summary"]
}
```

**编辑解读：** 注意 `intent` 使用了 `enum`。这将模型限制为一组预定义的类别，防止它发明新的意图，如 "customer_happiness_check"。同样，`maxLength` 防止摘要变成小说。这些约束是契约的一部分；违反它们意味着输出无效。

## 步骤 2：指定允许的值和缺失数据行为

一个经常被忽视的关键方面是模型应如何处理它没有的信息。如果没有明确的指示，模型往往会编造听起来合理的答案来填补空白。

您的契约必须明确说明当数据缺失时该怎么做。

*   **显式空值：** 指示模型如果无法确定字段，则返回 `null` 或 `""`。
*   **拒绝回答：** 对于敏感操作，指示模型返回特定标志，例如 `"confidence": "low"`，以便软件将请求路由给人工代理。
*   **默认值：** 避免要求模型猜测默认值。如果模型返回 null，让软件应用默认值。

**示例提示词指令：**
> “如果无法从文本中确定用户的紧急程度，请将 'urgency' 设置为 null。不要仅根据语气推断紧急程度。”

这减轻了模型的认知负荷，并确保软件收到关于数据质量的一致信号。

## 步骤 3：实施拒绝条件

一旦定义了契约，您的软件就需要一个验证层。该层将原始输出与模式和额外的行为规则进行检查。

拒绝条件应该是二元的：输出要么通过，要么失败。常见的拒绝触发因素包括：
*   **模式违规：** 缺少必填字段、数据类型错误或值不在允许的枚举中。
*   **安全违规：** 模型输出的内容触发了内部安全过滤器。
*   **长度违规：** 文本超过最大字符限制。
*   **格式错误：** 尽管被请求，但输出不是有效的 JSON/XML。

当发生拒绝时，软件不应简单地崩溃。它应该记录错误，可能使用修改后的提示词重试请求（例如，“请确保所有必填字段都已填写”），或升级至人工审核。

## 具体工作流：可逆测试

要实现这一点，请遵循以下工作流：

1.  **定义契约：** 在编写任何代码之前，写下确切的 JSON 模式和行为规则（缺失数据、枚举）。
2.  **构建提示词：** 将模式和规则嵌入系统提示词中。使用清晰、命令式的语言。
3.  **生成输出：** 向 LLM 发送请求。
4.  **验证：** 使用严格的解析器运行输出（例如，Python 中的 `pydantic`，JavaScript 中的 `zod`）。
5.  **处理结果：**
    *   **通过：** 继续执行业务逻辑。
    *   **失败：** 记录具体违规情况（例如，“字段 'urgency' 缺失”）。重试或回退。

**非官方实践示例：**
想象一个从电子邮件中提取日期的系统。
*   **契约规则：** `date` 必须是 ISO 8601 格式（`YYYY-MM-DD`）。如果未找到日期，返回 `null`。
*   **提示词：** “提取日期。返回 ISO 8601 格式。如果未提及日期，请将日期字段返回为 null。”
*   **验证：** 软件检查结果是否为有效的 ISO 日期或 null。如果模型返回“下周二”，验证将失败，系统将标记该电子邮件进行手动处理。

## 局限性与不确定性

虽然输出契约显著提高了可靠性，但它们并非万能药。

*   **模型能力：** 并非所有模型都支持严格的 JSON 模式或完美遵循复杂模式。较小或能力较弱的模型仍可能违反约束。
*   **提示词敏感性：** 契约的有效性在很大程度上取决于提示词工程。模糊的指令可能导致难以调试的细微违规。
*   **语义有效性：** 契约可以确保*格式*正确，但不能确保内容的*真实性*。如果模型因为误解上下文而提取了错误的日期，输出仍然会通过验证。
*   **延迟：** 添加验证和重试逻辑会增加应用程序的整体端到端延迟。

此外，来源材料表明，提示策略可能会影响输出的一致性，但不能保证完美无缺。开发人员必须假设始终需要一定程度的错误纠正。

## 何时使用

在以下情况下使用 AI 输出契约：
*   **数据完整性至关重要：** 您将 AI 输出直接馈送到数据库、API 或金融系统中。
*   **需要自动化：** 您希望最大限度地减少人工介入干预。
*   **需要结构化提取：** 您正在从不结构化文本中提取特定实体（日期、姓名、金额）。
*   **下游系统脆弱：** 您现有的软件无法优雅地处理意外格式或缺失字段。

## 何时跳过

考虑跳过严格契约的情况：
*   **目标是创意生成：** 撰写故事、诗歌或头脑风暴想法，其中期望具有可变性。
*   **探索性分析：** 您使用 AI 总结大型文档供人工审查，其中轻微的格式错误是可以接受的。
*   **低风险交互：** 用于随意对话的聊天机器人，偶尔出现幻觉是可以容忍的。
*   **资源限制：** 实施严格验证会增加开发开销。如果修复不良输出的成本很低，更简单的方法可能就足够了。
